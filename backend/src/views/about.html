About This Page

MoMA Acquisition Intelligence Platform

This page is served directly by the backend (/about) and summarizes architecture, implementation choices, limitations, and alternatives for the EAD assignment.

How the Application Works

The frontend (Next.js) sends HTTP requests to the Node.js/Express API. The API routes delegate to controller/service layers, where validation and business rules are applied, then data is persisted via Mongoose in MongoDB.

Client Layer

Next.js UI for CRUD testing, filtering, sorting, pagination, and acquisition workflows.

↓ HTTP / JSON ↓

API Layer

Express routes: /api/artworks, /api/users, /api/acquisitions.

↓ Mongoose ↓

Data Layer

MongoDB collections for artworks, users, and acquisition records/status history.

Technologies Used

Next.js (App Router) React Node.js + Express MongoDB + Mongoose CORS + Helmet + Morgan REST API Design

Implemented Functionality

  • Artwork CRUD with search, sort, and pagination.
  • User profile management (buyer/manager/admin roles without login tokens).
  • Acquisition tracking with state changes and purchase workflow constraints.
  • Rule enforced: an acquisition can only be marked acquired when current state is approved.
  • Responsive frontend that tests backend REST endpoints from browser.

Main Limitations / Weaknesses

  • No authentication/authorization flow yet (intentionally simplified for API testing).
  • Background image quality still depends on the source photo quality.
  • For very large datasets (10k+), additional indexing and query tuning may be needed.
  • Current UI is optimized for assignment demonstration, not full production hardening.

Alternative Approaches

  • Use Next.js API routes only (single runtime) instead of separate Express layering.
  • Use PostgreSQL for stricter relational constraints (especially purchases/orders).
  • Add JWT-based auth and role-based access control for enterprise security.
  • Add Redis caching for frequent search/filter result sets.
