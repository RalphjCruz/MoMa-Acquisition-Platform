About This Page

MoMAFlow

This page is served by the backend at /about. It documents the current assignment build, including how the application works, the technologies used, the main workflows, known limitations, and alternative approaches that could have been used.

General Explanation

MoMAFlow is a full-stack artwork marketplace application based on a Museum of Modern Art style catalogue. Users can browse artworks, search and filter the catalogue, manage artwork records, and use role-based workflows depending on whether they are acting as a buyer or manager.

The frontend sends HTTP requests to the backend API. The backend handles validation, business logic, authentication, and database operations. MongoDB stores the main application data such as artworks, users, credentials, and acquisition requests.

Client Browser Next.js UI pages, forms, filters, cart, and role-specific screens.
REST API Express routes receive requests such as GET, POST, PATCH, and DELETE.
Business Logic Validation, role checks, catalogue rules, and acquisition workflow handling.
MongoDB Stores artworks, users, credentials, carts, and acquisition records.

Architecture Overview

Frontend

Built with Next.js. It provides the user interface for browsing artworks, filtering the catalogue, logging in, registering, managing carts, and viewing role-based features.

API

Built with Node.js and Express. It exposes RESTful endpoints for artworks, users, authentication, and acquisition workflows.

Database

MongoDB is used with Mongoose models to structure and manage data for artworks, users, credentials, and acquisition records.

Technologies Used

The project uses a modern JavaScript full-stack structure with a separate frontend and backend. This makes the application easier to test, explain, and extend.

Next.js React Node.js Express MongoDB Mongoose REST API JWT Authentication HTML CSS JavaScript
  • Next.js/React: used to build the interactive client-side interface.
  • Node.js: used as the JavaScript runtime for the backend server.
  • Express: used to define clear REST routes and backend middleware.
  • MongoDB: used as the NoSQL database for flexible artwork and user data.
  • Mongoose: used to define schemas and interact with MongoDB more safely.
  • JWT: used to support login sessions and protect role-specific routes.

Role Workflows

  • Buyer: can browse the artwork catalogue, search/filter items, add artworks to a cart, submit pending purchase requests, and view the status of submitted requests.
  • Manager/Admin: can manage artworks using CRUD operations, review users, view pending acquisition requests, and approve or complete acquisition workflows.
  • Catalogue rules: artworks that are already in a buyer cart, already requested, or already acquired can be excluded from the available catalogue to avoid duplicate purchase actions.

Auth & Validation

  • JWT login and registration are used for authenticated access.
  • Role-based route protection separates buyer features from manager/admin features.
  • Registration requires username, email, role, password, and confirm password.
  • Password rules require at least 8 characters, one uppercase letter, and one special character.
  • Frontend validation improves user experience before data is sent to the backend.
  • Backend validation protects the API from invalid or incomplete requests.

REST Endpoints

The application is built around RESTful services. The main route groups are:

/api/auth, /api/artworks, /api/users, /api/acquisitions

  • Create: add new artwork records or acquisition requests.
  • Read: retrieve artworks, users, request statuses, and catalogue data.
  • Update: edit existing artworks and update acquisition statuses.
  • Delete: remove artwork records where permitted.
  • Search/filter/sort: supports easier catalogue navigation and testing.
  • Pagination: prevents the interface from trying to load the entire dataset at once.

Assignment Alignment

  • Uses a Node.js/Express backend server.
  • Loads and manages an artwork catalogue through backend services.
  • Stores application data using MongoDB and Mongoose models.
  • Implements RESTful API endpoints for client-server communication.
  • Supports artwork Create, Read, Update, and Delete operations.
  • Provides a client interface that allows users to test the main REST services.
  • Adds extra features such as role workflows, authentication, filtering, sorting, pagination, cart logic, and acquisition tracking.

Main Limitations / Weaknesses

  • Not fully production hardened: the application is suitable for assignment demonstration, but a real production system would need stronger logging, monitoring, deployment security, and error recovery.
  • Limited payment functionality: the buyer workflow simulates acquisition requests, but it does not integrate a real payment provider such as Stripe or PayPal.
  • Basic role model: roles such as buyer and manager are implemented, but a real enterprise system would need more detailed permissions, audit logs, and possibly multiple admin levels.
  • Dataset performance: for very large datasets of 10,000+ artworks, additional indexing, query tuning, and caching may be needed to keep search and filtering fast.
  • Limited testing coverage: the app can be manually tested through the interface and API routes, but a stronger version would include automated unit tests, integration tests, and end-to-end tests.
  • Simple UI state handling: the current interface is designed for clarity and demonstration. A larger system may need more advanced state management for carts, sessions, filters, and request updates.
  • Authentication could be stronger: JWT authentication is implemented, but a production system would also need refresh-token handling, account lockout, password reset, email verification, and stronger session security.
  • Concurrency issues: if two users try to request or acquire the same artwork at the same time, the system would need stricter transaction logic to guarantee that the artwork is only acquired once.
  • Deployment limitations: the assignment version is mainly built for local demonstration. A live hosted version would need environment variables, secure database hosting, HTTPS, and production build configuration.

Alternative Approaches

This project uses a Next.js frontend with a separate Express backend because it makes the REST API structure clear. However, the same application could have been built using other approaches.

  • Next.js API routes only: the backend logic could have been implemented directly inside Next.js API routes. This would reduce the number of separate project layers, but it may make the backend structure less explicit for demonstrating REST architecture.
  • PostgreSQL instead of MongoDB: a relational database could provide stronger structure for users, purchases, carts, and acquisition records. This would be useful if the project needed strict relationships and transactions.
  • Session-based authentication: instead of JWT, the app could use server-side sessions or a library such as Auth.js/NextAuth. This could simplify secure login handling, but may add more framework-specific setup.
  • GraphQL instead of REST: GraphQL could allow the frontend to request only the exact data it needs. However, REST is simpler to demonstrate and better matches the assignment requirement for RESTful endpoints.
  • Redis caching: frequent search and filter results could be cached using Redis to reduce database load. This would be more useful in a production-scale application with many users.
  • Cloud image storage: artwork images could be stored using services such as Cloudinary, AWS S3, or Firebase Storage. This would improve image delivery and reliability compared with basic local/static image handling.
  • Full deployment pipeline: the frontend could be deployed to Vercel, the backend to Render/Railway, and the database to MongoDB Atlas. This would make the app accessible online instead of only through localhost.
